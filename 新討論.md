CPU 瓶頸和 GPU 瓶頸的解法完全不同。
開 Chrome DevTools 的 Performance 分頁錄一段,
看時間軸主要卡在 Scripting 還是 Rendering/Painting;
或更快地直接印 three.js 內建的 `renderer.info.render.calls`/`.triangles`/`.points`,
看 draw call 數跟每幀處理的頂點數是否合理。這決定你該往哪個方向下刀。

**星星本身:通常是最大宗成本**
250 萬顆星如果每幀整包丟給 GPU、又是逐顆物件而非單一 buffer,幀率必死。
建議的管線大致如上圖,重點是三件事:

1. 全部星星只能是**一個** `THREE.Points` + 一個 `BufferGeometry`,
絕不要每顆星一個 mesh/sprite——draw call 數量本身就是頭號殺手。
2. 赤道座標→螢幕座標的轉換要丟給頂點著色器算,
不要在 JS 逐顆逐幀算。
你的 implementation_plan.md
裡月球那段其實已經在用 `eqToHoriz`/`lookAz`/`lookEl`/`focalLen`
這組 uniform 在 GPU 端轉換——方向是對的,
但要確認同一套邏輯有沒有真的套用在星星本身,
不要變成「月球走 GPU、星星卻在 JS 端跑三角函數」。
3. healpixjs 的價值不該只用在查詢,
該拿來做真正的 LOD:依目前視角只挑出視錐內的 tile,再依縮放等級設星等門檻
(縮小時只留較亮的星,放大再逐步放寬),每幀真正上傳給 GPU 的永遠是一小撮子集,
而非全部 250 萬顆。

閃爍動畫同理,別每幀在 CPU 改 buffer,
傳一個 `uTime` uniform 配上每顆星建置時就算好的隨機種子 attribute,
shader 裡用 `sin(uTime * speed + seed)` 算亮度起伏,CPU 完全不碰資料。

**像素成本與全域設定**
星星的圓形光暈如果在 fragment shader 逐像素算距離衰減,
4K、多螢幕情境下很吃 fill-rate;
改成一張預烘焙的柔光點貼圖,
additive blending 直接貼上,成本幾乎是常數。
銀河霧光你已經有 `milkyway.png`,直接當 equirectangular skybox 疊圖,
別即時算大氣散射。
另外:`renderer.setPixelRatio(Math.min(window.devicePixelRatio, 1))`
(桌布不需要 2x/3x 解析度)、antialias 改用後製 FXAA 或乾脆關掉、
確認沒有多餘的 shadow map——這幾個常被忽略但影響很大。

**檢查是不是同時跑了兩套引擎**
`webgl_engine.js`(Three.js)跟整包
`stellarium-web-engine-master`(C 編譯成 WASM)同時存在你的 repo 裡。
如果兩者現在都真的在跑各自的 render loop,等於兩個 WebGL context 同時搶同一張顯卡,
很容易讓幀率直接腰斬,而且不容易從畫面上看出來。先確認只有一套真正在渲染。

**桌布引擎本身的節流機制,別重造輪子**
Wallpaper Engine 那邊,
官方文件說明了如何讀取並套用使用者在 Performance 設定分頁調整的 FPS 上限,
並建議直接沿用這個由使用者設定的上限,
而不是自己在桌布裡另外寫死一個,
做法是監聽 `wallpaperPropertyListener.applyGeneralProperties`,
拿 `properties.fps` 節流你的 `requestAnimationFrame`。Lively 這邊,
不論哪種桌布類型,只要有全螢幕應用程式或遊戲在跑,Lively 都會完全暫停桌布播放,
但官方 wiki 特別提醒暫停時只是畫面不再繪製,
網頁裡的 JavaScript 其實還是會繼續執行,
所以你該自己監聽 `livelyWallpaperPlaybackChanged` 拿 `IsPaused`,是 `true` 時主動停掉自己的 `requestAnimationFrame`,別讓 JS 空轉燒 CPU。另外Lively 支援透過 LivelyProperties.json 讓它自動產生設定選單,網頁裡用 livelyPropertyListener(name, val) 監聽使用者調整,可以開一個「星星密度/畫質」滑桿讓弱機自己調低。
還有一點,Wallpaper Engine 預設把每個獨立顯示器當成獨立實例處理,
螢幕數量增加,GPU 跟記憶體負擔也會跟著疊加——多螢幕天生要付出這個成本,
不是程式 bug,值得先在說明文件提醒使用者。

**回頭想一下:裝飾用桌布,還是精確模擬**
Tycho-2 全量 250 萬顆,但肉眼在理想暗空下看得到的星大約只到 6.5 等,
全天球加起來約 9000 顆,
單一地平面通常只看得到幾千顆——比 250 萬少了兩三個數量級。
如果定位是美術裝飾,「一直在畫的那一層」限制在較亮星等就好,
Tycho-2 全量留給未來的縮放/查詢功能,視覺上幾乎沒損失,頂點數卻直接砍掉兩三個數量級。

**藝術/演算法參考**

你的 repo 已經內嵌了 Stellarium Web Engine,
這其實是離你最近的參考——它處理巨量星表分塊渲染、大氣散射、地平線遮蔽的做法已經是業界公認成熟的方案,
直接翻它的 C 原始碼看它怎麼分塊、怎麼定 LOD,
會比重新發明一套更快。
你用的 healpixjs 剛好對應到天文可視化領域的標準協議 HiPS(Hierarchical Progressive Survey)——Aladin Sky Atlas
就是這套做法的代表實作:把整個天球依 HEALPix 切成金字塔狀的多解析度 tile,
使用者縮多遠就載入對應解析度的 tile。
這跟你專案的依賴選擇根本是天作之合,值得直接參考它的分塊策略,
而不是自己重新設計一套。
美術風格上,SpaceEngine 在超大尺度渲染(銀河霧光、恆星光暈、大氣散射的漸層處理)上做得很細膩,
尤其是「近看是一顆恆星、拉遠變成星野裡一個光點」這種跨尺度視覺過渡的手法,
如果你的桌布之後想加縮放/導覽功能,這是很值得借鏡的美術方向。
