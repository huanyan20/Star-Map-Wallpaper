 `frameState.js`(決定天空底色)、`render2D.js` 的 `drawHorizonGlow`、還有太陽本體的 `sky.js` shader,拼出目前「夕陽」到底是怎麼做的。

**你已經有的東西,其實不少**:

- 太陽圓盤本身會在低仰角時變紅變柔和(`sky.js` 裡 `sunColor = mix(橘紅, 米白, altFactor)`)
- 太陽周圍有一圈依高度變色的放射狀光暈(暖黃→橘→深紅→紫,`drawHorizonGlow` 的 `sGrad`)
- 反日點(anti-solar point)甚至做了「維納斯帶」(Belt of Venus)——粉紫光帶疊在地球影子上,這是很多業餘天文渲染都懶得做的細節,你有做
- 大氣 shader 裡也有真正的 Rayleigh/Mie 物理散射

**但缺一塊很關鍵的東西**:決定整片天空底色的 `frameState.js`(`topRGB/midRGB/horRGB`)只有三組顏色——夜晚、暮光、白天——而且全部都是藍黑色系,完全沒有橘紅色。我把這條色彩曲線實際算出來畫成圖表:你會看到「目前」那條線從夜晚的微暖一路單調降到白天的冷藍,在太陽高度 0° 附近完全沒有回暖的波峰;而「建議」那條線在同樣位置會有一個明顯的暖色峰。換句話說,你所有那些漂亮的太陽光暈、維納斯帶,都是疊在一片本質上偏冷的天空「畫布」上——這也是為什麼即使做了這麼多細節,整體觀感可能還是覺得夕陽「不夠」的根本原因:底色從來沒真的變橘過。

另一個放大這個問題的地方:物理大氣的散射結果雖然算得對,但在 `skyFragment.frag.glsl` 裡只用 `atmosphereBlend * 0.18` 的權重混進最終顏色,等於物理算出來的暖色被稀釋到只剩不到兩成,大部分顏色還是來自那條偏冷的底色曲線。

**具體怎麼修**

在 `frameState.js` 加一個真正的「夕陽色階」,用鐘形函數(以高度角 0° 為中心)把暖色混進 `horRGB`(可以也小幅混進 `midRGB`):

```js
const sunsetHor = [225, 120, 65]; // 夕陽橘,可自行微調
// ...算完現有的 lerp3 三段之後:
const sunsetWeight = Math.exp(-Math.pow(bgSunAlt / 8, 2)) * 0.75; // 高度角±8°內明顯,峰值在 0°
bgCache.horRGB = lerp3(bgCache.horRGB, sunsetHor, sunsetWeight);
```

`8` 是峰寬(調大會讓暖色持續更久,調小會更集中在太陽貼地平線那一刻),`0.75` 是峰值時的混合強度。真實的日落其實不對稱——日落前的暖化比日落後的餘暉散得慢一些——如果想更講究,可以把 Gaussian 換成左右不同寬度的版本。

同時建議把 shader 裡 `atmosphereBlend * 0.18` 這個係數,在 `|sunAlt| < 8°` 這個窗口內動態拉高(例如到 0.4~0.5),讓物理散射的暖色也能透出來,而不是全靠 CPU 端這條曲線硬做。

**美術上的參考點**

真實/好看的夕陽通常是這樣分層的:天頂深藍 → 中段淡青/米黃 → 地平線橘紅/鮭魚粉 → 太陽本體金白色核心。你現在的太陽光暈、維納斯帶其實已經抓到這個邏輯,只差把「地平線」那一層的底色跟著一起暖起來,整張圖才會連貫,而不是一片藍天上貼了一顆暖色貼紙。

還有一個低成本但很加分的細節:你已經有高雄天際線剪影(`getProceduralSkyline`),太陽貼近地平線時,可以讓剪影的邊緣加一條極細的暖色逆光描邊(classic rim-light 技巧),會讓整個畫面更有「攝影感」。
