直接讀了 `milkyway.png` 左右兩欄像素,平均差異只有 ~1.6(滿分255),銀道平面那一列(y=512)左邊 `[57,58,60]` 右邊 `[52,56,65]`,差距也不大。所以來源圖片本身不是主因。

真正的問題出在 `setupMilkyWayGlow` 那顆球體的 UV 重新賦值:

```js
const sphereGeo = new THREE.SphereGeometry(1, 256, 128);
...
for (let i = 0; i < posAttr.count; i++) {
  ...
  const [u, v] = eqToGalUV(gx, gz, gy);
  uvAttr.setXY(i, u, v);
}
```

`THREE.SphereGeometry` 原生產生的球體,其實本來就在自己的經度接縫處把頂點複製了一份(同一個 3D 位置,兩份頂點,一份 u=0、一份 u=1),這是所有 UV 球體的標準做法,目的就是讓接縫兩側可以各自帶不同的 u 值,GPU 插值才不會穿過整張圖亂繞。

但這段迴圈是拿每個頂點的「3D 位置」重新算銀經算出 u——而 `atan2` 是位置的determinstic函數,同一個 3D 位置永遠算出同一個 l、同一個 u。也就是說,原本那對「同位置、不同 u(0跟1)」的複製頂點,現在被這個迴圈雙雙覆寫成同一個 u 值,原生自帶的接縫複製等於被抹掉了。結果就是:任何一個三角形,只要它的三個頂點剛好橫跨銀經 ±180° 這條線(比如 u 值是 0.98、0.99、0.02),GPU 會直接線性內插這三個 u 值——從 0.99 插到 0.02 是走「經過 0.5」這條近路,而不是繞過 1.0→0.0 的正確路徑,等於整張貼圖在這一圈三角形帶狀區域裡被反著、快速地掃過一遍。這就是螢幕上看到的撕裂/分岔——不是頭尾顏色對不上,是頭尾中間那圈三角形的 UV 插值方向錯了。這個 bug 只存在於 `mw_glow.png` 這顆球(setupMilkyWayGlow),粒子系統本身沒有這個問題,因為粒子是生成時就採樣好顏色,沒有跨三角形的 UV 插值這回事。

正規修法是做「接縫分割」:遍歷每個三角形,檢查它三個頂點的 u 值有沒有跳躍超過 0.5,有的話把那個(那些)頂點複製一份,複製體的 u 值加或減 1.0 讓插值方向連續,再把該三角形的索引指到複製體上,而不是共用原本那個頂點。這是標準做法,但要動索引緩衝區,稍微麻煩。

如果想先有個快速緩解:把 `eqToGalUV` 裡的經度接縫刻意轉到銀經 180°(銀河反中心方向)——那裡本來就是整條銀河帶最暗、結構最少的地方,就算接縫沒完全修好,視覺上也不會被注意到,可以先擋著,之後再排時間做真正的頂點分割。
